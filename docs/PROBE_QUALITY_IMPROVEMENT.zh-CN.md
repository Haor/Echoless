# 延迟 Probe 质量判定改进计划

状态：第一阶段已实现并通过实际使用验证；第二阶段取消

适用范围：Echoless 1.2.x 后续开发

## 问题结论

旧实现只用标准差和首尾漂移推导 `stable`，麦克风静音、底噪、环境讲话或错误路由仍可能
产生稳定但错误的结果并触发自动回填。第一阶段已经把信号有效性、事件数量、延迟一致性和
自动回填统一到后端质量结论。产品保持这套简单质量门，不再增加频率识别等第二阶段机制。

## 第一阶段当前实现

实现位于 `crates/echoless-cli/src/probe_delay.rs`、`app/src/probeQuality.ts` 和
`app/src/pages/AdvancedPage.tsx`：

1. CLI 默认生成 12 次、每次 70 毫秒的蜂鸣，并以 0.5 毫秒 RMS 能量包络分析 reference
   与 mic。`--beeps` 最少为 2。
2. 只有正相关且 `corr > 0.15` 的事件能成为延迟候选；命中正负 250 毫秒搜索边界的候选
   被拒绝。
3. reference 检测事件与有效延迟都必须达到预期蜂鸣数的至少三分之二并向上取整，且不得
   少于 2 个；默认是 12 次中至少 8 次。
4. 候选延迟先取最密集的 10 毫秒宽度簇。该簇还必须满足标准差不超过 5 毫秒、首尾漂移
   绝对值不超过 10 毫秒，最终建议使用簇内中位数。
5. reference 必须同时达到 `-45 dBFS` 绝对电平和 10 dB 事件/背景能量对比；mic 必须
   同时达到 `-55 dBFS` 和 4 dB 事件/背景能量对比。绝对电平作为无法仅靠时序对比排除
   极弱信号时的兜底硬门。
6. 后端输出 `valid`、`uncertain`、`invalid` 三态质量以及明确原因。非 `valid` 结果的
   `event_lag_mean_ms`、`event_lag_stddev_ms`、`event_lag_drift_ms` 和
   `recommended_near_delay_ms` 都为 `null`。
7. GUI 只在 `quality == valid` 时自动回填；`uncertain` 和 `invalid` 是正常完成的测量，
   CLI 返回退出码 0，但 GUI 不修改现有延迟配置。

## 第一阶段已关闭的缺陷

- 空集合或单个偶然事件不再能伪造零标准差、零漂移的稳定结果。
- 麦克风静音、低电平或事件/背景对比不足会返回 `mic_signal_missing`。
- 负相关峰、少量相关峰和搜索边界命中不会产生有效建议。
- 多数一致、少数离群时，以最密 10 毫秒簇和中位数保留主要延迟。
- macOS 与 Windows 的 GUI 回填共用后端 `quality` 结论，不再各自推导稳定状态。
- 已补充静音、噪声、单事件、负相关、边界、离群簇和能量对比等确定性回归测试。

能量包络不验证四频蜂鸣身份。这是有意保留的简化：第一阶段已经能够拒绝静音、弱信号、
少量事件和不一致结果，继续增加频率识别对当前小型工具的收益不足以抵消复杂度。

## 改进原则

- 后端负责产生唯一的质量结论，GUI 不再根据零散字段自行推导 `stable`。
- 测量有效性与测量一致性分开判断。先证明信号可信，再评价延迟是否稳定。
- 无法证明有效时不得自动回填，但仍可保留诊断字段供维护者分析。
- `global_lag_ms` 只作为诊断信息，不再作为零有效事件时的回填候选。
- macOS 与 Windows 使用同一质量门；平台差异只保留在最终参数映射中。
- 阈值由确定性测试保护，并允许根据后续真实故障样本做小幅校准；不预先扩展算法层级。

## 第一阶段：阻止错误回填（已实现）

第一阶段保留现有包络相关算法，增加保守的质量门和统一契约。

### 后端质量结果

Probe 结果现在包含以下权威质量字段：

```text
quality: valid | uncertain | invalid
quality_reasons:
  - ref_signal_missing
  - mic_signal_missing
  - insufficient_reference_events
  - insufficient_valid_lags
  - weak_correlation
  - inconsistent_lags
  - lag_at_search_boundary
```

CLI 与 GUI 消费同一结果。当前硬门和候选阈值以“第一阶段当前实现”一节为准；阈值可以在
真实录音校准后调整，但 JSON 字段和三态语义应保持稳定。

### 回填纪律

- 只有 `quality == valid` 才允许写入 `near_delay_ms` 或 `initial_delay_ms`。
- `uncertain` 和 `invalid` 只显示原因并建议重试，不修改现有配置。
- GUI 的稳定文案直接来自后端质量结果；`Ref OK`、`Mic OK` 与最终状态保持一致。
- AEC3 初始延迟和 macOS 近端延迟都在同一个质量门之后执行。

## 第二阶段：不实施

不引入频率身份识别、窄带匹配滤波或更多置信度层级。除非以后出现可重复、影响实际使用且
无法通过现有阈值修正的故障样本，否则 Probe 保持第一阶段结构。

## 验证状态

第一阶段已有以下自动化证据：

- `cargo test -p echoless-cli --locked`：143 项通过，覆盖静音、确定性噪声、错误路由、
  单事件、少量事件、负相关、搜索边界、离群簇以及 reference / mic 能量对比门。
- 前端 Vitest：124 项通过，覆盖三态展示、原因映射以及仅 `valid` 自动回填。

第一阶段已完成实际使用验证。后续只在收到可重复失败样本时调整现有阈值，不把扩展算法设为
发布前置条件。

### 已覆盖的确定性合成场景

- reference 正常、mic 全零：必须 invalid，不回填。
- reference 正常、mic 仅低电平白噪声：必须 invalid。
- reference 正常、mic 只有一个匹配脉冲：不得 valid。
- reference 只检测到少量蜂鸣：不得 valid。
- 负相关峰高于正相关峰：不得把负相关作为有效事件。
- 最佳延迟等于正负 250 毫秒边界：必须 uncertain 或 invalid。
- 多数事件一致、少量离群：稳健中位数应保持正确，离群事件不得拉偏建议值。
- 已知正负延迟并叠加不同强度噪声：在规定信噪比范围内保持正确。

## 后续顺序

1. 保持当前 JSON 质量契约和只在 `valid` 时回填的纪律。
2. 只有收到可重复的真实失败样本时，才在现有质量门内校准阈值。
3. 不实施第二阶段频率感知匹配。

## 验收标准

- 静音、错误路由、少量事件和纯噪声不会显示 stable，也不会修改配置。
- 有效 Probe 在受支持平台能给出可重复的延迟建议。
- CLI、GUI 状态文案和自动回填使用同一质量结论。
- 无有效事件时不会用全局相关结果伪造零标准差、零漂移的稳定结果。
- 回填前的所有质量门都有对应回归测试。

## 不在本轮范围

- 改写设备采集、系统音频参考或虚拟音频路由。
- 为了保留旧错误行为而增加兼容分支。
- 在没有样本证据前承诺固定的事件数、相关性或信噪比阈值。
- 把 Linux 硬件验证设为该改进的发布阻塞项。
- 增加频率身份识别、窄带匹配滤波或新的质量层级。
